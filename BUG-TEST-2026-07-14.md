## Teacher portal / co-op server test (`coop/`)

All `coop/` files read completely and pass `node --check`; `duel-bank.json` is a valid 113-question
array. The server was booted on a test port and exercised with a real HTTP client (no Redis for the
endpoint pass; an in-memory Redis stub for the account/PIN pass).

### Endpoints & teacher auth — 29/29 pass
- `/host`, `/join`, `/privacy`, `/__coop/client.js`, `/__coop/ping` all serve; baseline security
  headers present (`X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN`,
  `Referrer-Policy: no-referrer`).
- `/__coop/create` returns a join code **and** a per-room `teacherKey`.
- **Teacher gating works:** world-rename, `tq/create`, `tq/decide`, `tq/end`, and `kick` all return
  **403 without the key** and succeed **with it**.
- **No answer leaks:** duel views (pending and active) carry **no `best`** on the wire; the student
  class-question view omits the `correct` index.
- **Duel is server-authoritative:** questions come from the server bank, scoring is done server-side,
  and the star delta is zero-sum.
- Student class-question answer marks correctness server-side and rewards correctly.
- Bad JSON → 400; **oversized body (>1 MB) settles in well under 5 s** (no hang); unknown route → 404.

### Accounts / PIN (in-memory Redis stub) — 10/10 pass
- New account creation; correct-PIN re-signin; wrong-PIN rejected.
- Save requires the correct PIN; a wrong-PIN save is denied and does **not** overwrite saved data.
- Saved data round-trips on next sign-in.
- **Brute-force lockout** triggers (HTTP 429) after repeated wrong PINs and blocks even the correct
  PIN during the lockout window; a different account is unaffected.

### Dashboard / client wiring
- `dashboard.html` inline JS and `join.html` parse cleanly.
- The dashboard persists the `teacherKey` per room in `localStorage` and sends it on every teacher
  action (world / tq create / tq end / tq decide).
- The injected client never references the teacher key (students cannot perform teacher actions),
  shows **"Not in a class yet"** before joining, and links the privacy notice with
  first-name-only guidance. Privacy link + tip also present on the dashboard.

### Verdict: teacher portal is clean — no bugs found.
- **Design note (not a bug):** the shared **MAIN world** (the default room, which has no teacherKey)
  intentionally leaves teacher actions open for legacy anonymous use — so in that open world anyone
  could rename it or post a class question. For classroom use teachers should always click **Start a
  Class** (which creates a keyed, gated room). Worth a one-line note in the host instructions if you
  want to be explicit about it.
