# Full Bug Test — The Policy Game (2026-07-13)

Multi-agent test run. Three headless QA agents each drove simulated users in parallel, plus a live browser pass in the main session.

- **Agent 1 — Game logic:** simulated a full player under jsdom (all 493 policy questions × every choice, currency/purchases, bank, save/load, tiers).
- **Agent 2 — World geometry:** audited colliders, roads, river, and interaction spots across 18 ownership/tier states.
- **Agent 3 — Co-op layer:** static review of the server/client plus a light live load of multiple simulated clients against the deployed server.
- **Main session — Live UI:** signed in as TestKid on the live site, booted the game, opened City Builder, checked meters and buy-states, watched the console.

Overall: the single-player game is very robust — **no crashes, no NaN, no economy exploits, no collider overlaps.** The real issues are (1) a content regression I introduced with the last shop-spacing change, and (2) a cluster of co-op cheating/abuse holes that matter before classroom publication.

---

## Priority fixes

### 1. MEDIUM — Only 8 of 12 shop kinds ever appear (regression from the shop-spacing change)
Found independently by the logic and geometry agents. The east shop rows use `row(y, 2895, 3500, 400, …)` across 4 rows. With origin 2895 and step 400, only two lots fit per row (2895, 3295), so 2×4 = **8 storefronts**. `nextShop()` hands out the first 8 kinds only, so **florist, diner, music, and hardware never spawn anywhere in the game.** The nearby code comment ("3 wider slots × 4 rows = all 12 shop kinds appear") is now stale.

This is a side effect of the "one shop per block, more spaced out" change you asked for — the wider spacing dropped the row capacity. No gameplay gate requires all 12 (shops are cosmetic + mini-game entries), so nothing is uncompletable, but four shop designs are dead content the player never sees.

Fix options: add a 5th shop row for the remaining four kinds, or slightly tighten spacing so three fit per row, or accept 8 and trim the shop-kind list to match. Recommend deciding based on whether you want all 12 designs visible.

### 2. HIGH (co-op) — No teacher authentication
Any request that knows the 4-char room code can create/rename the world, kick students, and post/end class questions. There is no teacher token or session (`coop-server.js` create/world/kick/tq routes). Confirmed live: world rename, arbitrary kick, and class-question creation all succeeded from a plain student-level request.

### 3. HIGH (co-op) — Students can award themselves stars
`/__coop/tq/decide` marks a class question `awarded:true` for any `{room,id}` with no teacher check; the client then credits the stars. A student can grant their own reward. Confirmed live.

### 4. HIGH (co-op) — Duel answer key comes from the challenger
The challenger sends the questions **and** their correct answers; the server scores against that verbatim. A malicious challenger can craft a guaranteed win and take the wagered stars. Confirmed live (attacker won +1000, victim −1000).

These three co-op issues share one root cause: the server trusts client-declared identity, scores, and roles. The single highest-leverage fix is a teacher secret returned by `/__coop/create` and required by `world`/`kick`/`tq/*`/`decide`, plus server-side duel scoring.

## Co-op — additional findings

- **MEDIUM — Spoofable economy/identity.** `/__coop/sync` stores the raw client object; `id`, `stars`, `score`, `name` are all client-declared. Any client can impersonate another id or publish an arbitrary leaderboard score (test set stars to 88,888 instantly).
- **MEDIUM — Unbounded room growth.** `pruneAll()` runs only when a dashboard polls `/__coop/state`; there is no timer. Sync/create both make rooms, so without a dashboard open the `rooms` map can grow unbounded. Add a `setInterval(pruneAll, …)`.
- **MEDIUM — Global shared default world.** Everyone with no class code lands in `MAIN` and streams into one shared world worldwide. Consider requiring a code on the public deployment.
- **LOW — Oversized body → 502.** A ~1.2 MB POST to `/__coop/sync` returns a Render 502 (~250 ms); `readBody` destroys the socket but the handler still awaits an `'end'` that never fires, leaving a dangling promise. Server survives (ping still 200). Fix: resolve/reject on `aborted`/`close`.
- **LOW — Duel double-book TOCTOU** (the check straddles an `await`) — theoretical, not reproduced live.
- **LOW — 4-char codes can be reissued after prune**; `normCode` accepts 6 chars while codes are 4 (cosmetic mismatch).
- **Verified working (co-op):** room create/join, per-room isolation, live state propagation, clean leave + reconnect, graceful 400s on malformed JSON / missing id, no-crash on weird value types, Redis-off degrades cleanly, PIN auth uses sha256 + constant-time compare + per-IP brute-force throttle, dashboard escapes untrusted names/answers (no stored XSS).

## Game logic — findings (all low severity)

- **LOW/MEDIUM — Mini-game turn burned on entry.** `openMiniGame()` sets `S.mini[type]=true` on entry, before you play; backing out still consumes that term's reward. May be intended (a comment suggests so), but an accidental walk-in silently costs the reward.
- **LOW — `mgResult` has no internal per-term guard.** Not reachable via UI (entry is gated), but the reward function itself double-awards if called twice directly.
- **LOW — `clamp` doesn't sanitize NaN.** Normal play can't produce it (all effects are integers), but a corrupt/injected stat would render "NaN%".
- **LOW — `pickChoice('Z')` throws** on an invalid letter; UI only passes valid letters.
- **LOW — `activateProfile` doesn't default `P.stars`.** Harmless (all reads use `P.stars||0`) but inconsistent with the other defaulted fields.

### Verified clean (logic)
All 493 questions × every choice keep stats in 0–100 with no NaN; no double-spend, no free buys, no negative balances across houses/districts/dev/attractions/emotes/boxes; town bonus caps at 18/meter and clamps to ≤100; save/load round-trips and self-heals on 8 malformed-save cases; 5-term stability with no NaN; bank interest capped (no infinite minting); score floored ≥10; tier bands inclusive with no off-by-one.
