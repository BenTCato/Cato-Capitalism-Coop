# Bug Test — 2026-07-21 (multiplayer classroom simulation)

A ~20-minute live bug hunt against the **deployed** build
(`https://cato-capitalism-coop.onrender.com`), designed to mimic a real class:
**four simulated students** (separate Chromium browser contexts, separate game
accounts, all in one shared class code) playing concurrently, plus a **teacher**
polling the dashboard, for **15 rounds** of ~30s each. Ran alongside a static
source audit of `FreedomFounder_v4.html` and the `coop/` server + client.

**Bots and personas**
- `speedy` (Speedopolis, grade 6) — walks fast, answers rapidly, double-clicks choices and mashes SPACE (race testing).
- `tycoon` (Bankersburg, grade 10) — spams Build/Bank/Store/Home; deposits/withdraws odd amounts (0, negative, more than balance); unboxes twice fast.
- `social` (Friendlyland, grade 8) — emotes and initiates duels with other bots, then answers duel questions.
- `chaos` (nation name = an XSS payload, grade 12) — mashes Escape, opens/closes every overlay, enters bad class codes (`💥💥💥`), reloads mid-question.
- `teacher` — reads `/__coop/state`, posts a Class Question every 3rd round, and cross-checks each bot's on-screen stats against the dashboard.

## Result: 2 confirmed bugs (1 High, 1 High), 1 minor cosmetic, strong overall stability

Across 15 rounds the game **never crashed, never threw an uncaught JS error in
normal play, never produced a negative/NaN star or bank balance, and never
drove a stat outside 0–100.** Bad class codes, emoji codes, rapid overlay
toggling, double-clicks, and mid-question reloads were all handled gracefully.
The issues below are specific and reproducible.

---

### 🔴 HIGH 1 — Duel reward can be duplicated by reloading (star exploit)
**Where:** `coop/coop-server.js` (duel result lingers 60s, no "credited" flag) + `coop/coop-client.js` `applyResultOnce()` (guard is in-memory only).

A finished duel stays on the server for 60 seconds and `duelView()` returns the
authoritative star `delta` on **every** sync during that window. The only thing
stopping the winner from banking the pot twice is the client's in-memory
`applied{}` map — which is wiped by a page reload, while `coopId`/`coopRoom`
persist in sessionStorage and `sync()` runs every 160ms. So a duel **winner who
presses F5 within 60s re-applies `+wager` again** (and `duelWins`/`duelStarsWon`
inflate too). Reloading repeatedly multiplies winnings.

**Verified live:** after a settled 500⭐ duel, three consecutive syncs as the
winner each returned `delta = +500`. The teacher-question flow already solves
this correctly with a server-side `credited` flag; duels need the same.

**Fix direction:** mark the duel's per-player payout `credited` server-side the
first time it's reported (mirroring `tqStudentView`), and/or zero the delta once
sent, so a reload can't re-award it.

---

### 🔴 HIGH 2 — Nation name is not escaped (stored XSS)
**Where confirmed:** `FreedomFounder_v4.html:3028` — `btn.innerHTML = 'Found ' + nationName + ' & Begin →'` injects the raw nation name into the DOM.

A nation name of `<img src=x onerror=…>` executes script. Reproduced
deterministically: the payload runs inside `#begin-btn` on the player's own
screen. During the multi-bot run the `chaos` bot's malicious name **also fired
in other students' browsers 4 times across 15 rounds** — an intermittent
cross-user path — so this is not purely self-XSS; a crafted nation name can run
code on a classmate's (or the teacher's) machine.

Most name sinks *are* escaped (`escapeXml` on co-op nameplates, duel dialogs,
emote/chat bubbles; `esc()` on the dashboard; `textContent` on the newspaper
masthead and town/header labels), which is why it's intermittent rather than
constant — but at least one render path plus the begin button are unescaped.

**Fix direction:** sanitize the nation name at the source (strip/escape `<>&'"`
in `customNationName()`), escape it at the `#begin-btn` sink, and — defense in
depth — have the co-op server strip HTML from `name` before storing it so no
client can ever receive a payload.

---

### 🟡 LOW 3 — Dashboard health bars lag the student's live screen by a few points
**Where:** dashboard reads a `/__coop/state` snapshot that is up to ~9s old; the student HUD is live.

The teacher cross-check flagged small, **bidirectional** gaps (e.g. HUD 68 vs
dashboard 65; HUD 65 vs dashboard 68) — always ≤5 points, in both directions.
This is sync latency, not the earlier stat-source bug (that was fixed; both now
read `dispStats()`), and it self-corrects within a couple of seconds. Cosmetic,
but if desired the dashboard could show a "updated Ns ago" note or poll faster.

---

## What was exercised and held up (no bugs found)
- **Economy invariants:** bank deposit/withdraw with 0, negative, and over-balance amounts never produced a negative or NaN balance; stars never went negative; store double-unboxing refunded correctly.
- **Overlays & input races:** rapid Escape spam, opening/closing Build/Bank/Store/Home/School/minigame repeatedly, and double-clicking choices/Next did not stack broken state or crash.
- **Join flow:** invalid class codes (`💥💥💥`, wrong codes) were rejected cleanly; joining the right code mid-session worked.
- **Reloads:** reloading mid-question and re-entering the game restored the town without error.
- **Duels:** binding rules held — an active duel could not be cancelled by either side; a walk-out forfeited the pot to the player who stayed (confirmed in the prior duel test). *(The reload dup above is a separate, reward-crediting bug, not a "backing out" bug.)*
- **Teacher:** dashboard listed all 4 cities; Class Question create/post succeeded.

## Method notes
- Rig: `outputs/bugbot.js` (Playwright + headless Chromium, 4 contexts/round, persistent per-persona storage, error/console/network/invariant logging to `findings.jsonl`, screenshots on error).
- Isolation harnesses `findxss*.js` / `dupcheck.js` pinned the two High findings to exact sinks.
- 15 rounds run; 19 diagnostic screenshots captured; 0 uncaught errors logged at any round-end.
